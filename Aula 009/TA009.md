## 1. Conceitos Fundamentais

### Por que Kubernetes?

Imagine que você tem uma aplicação que precisa rodar em centenas de servidores, com milhares de containers, atualizações constantes, e requisitos complexos de rede e armazenamento. O **Docker Swarm** é excelente para cenários simples, mas para ambientes enterprise complexos, você precisa do **Kubernetes**.

**Kubernetes** (K8s) é a plataforma de orquestração de containers mais poderosa e amplamente adotada do mundo, originalmente desenvolvida pelo Google e agora mantida pela Cloud Native Computing Foundation (CNCF).

### Kubernetes vs Docker Swarm

| Aspecto | Docker Swarm | Kubernetes |
|---------|--------------|------------|
| **Complexidade** | Simples | Complexo |
| **Curva de Aprendizado** | Baixa | Alta |
| **Recursos** | Básicos | Avançados |
| **Ecossistema** | Limitado | Vasto |
| **Adoção Enterprise** | Menor | Dominante |
| **Networking** | Simples | Muito flexível |
| **Storage** | Básico | Avançado |

### Arquitetura do Kubernetes

#### Control Plane (Plano de Controle)
O **Control Plane** é o "cérebro" do cluster, composto por:

**API Server**: Ponto de entrada para todas as operações
- Expõe API REST do Kubernetes
- Autentica e autoriza requisições
- Valida e processa objetos K8s

**etcd**: Banco de dados distribuído
- Armazena todo o estado do cluster
- Backup crítico para recuperação
- Consenso via algoritmo Raft

**Scheduler**: Decide onde colocar Pods
- Analisa recursos disponíveis
- Considera constraints e affinity
- Otimiza distribuição de workloads

**Controller Manager**: Mantém estado desejado
- Monitora objetos do cluster
- Executa loops de reconciliação
- Garante que estado atual = estado desejado

#### Worker Nodes (Nós de Trabalho)
Os **Worker Nodes** executam as aplicações:

**kubelet**: Agente do Kubernetes no nó
- Comunica com API Server
- Gerencia Pods no nó
- Reporta status e métricas

**kube-proxy**: Proxy de rede
- Implementa Services
- Load balancing entre Pods
- Regras de iptables/IPVS

**Container Runtime**: Executa containers
- Docker, containerd, CRI-O
- Gerencia ciclo de vida dos containers

### Objetos Fundamentais do Kubernetes

#### Pod: A Menor Unidade
Um **Pod** é a menor unidade deployável no Kubernetes:
- Contém um ou mais containers
- Compartilha rede e volumes
- Sempre co-localizados no mesmo nó
- Efêmero (pode ser criado/destruído)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: meu-pod
spec:
  containers:
  - name: nginx
    image: nginx:1.21
    ports:
    - containerPort: 80
```

#### ReplicaSet: Garantindo Réplicas
**ReplicaSet** garante que um número específico de Pods esteja rodando:

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
```

#### Deployment: Gerenciamento Declarativo
**Deployment** é a forma recomendada de gerenciar aplicações:
- Gerencia ReplicaSets automaticamente
- Rolling updates e rollbacks
- Histórico de versões

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
```

#### Service: Descoberta e Load Balancing
**Service** expõe Pods para comunicação:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
  type: ClusterIP  # ClusterIP, NodePort, LoadBalancer
```

### Tipos de Services

#### ClusterIP (Padrão)
- Acesso apenas interno ao cluster
- IP virtual interno
- Load balancing entre Pods

#### NodePort
- Expõe serviço em porta específica de cada nó
- Acesso externo via <NodeIP>:<NodePort>
- Range padrão: 30000-32767

```yaml
spec:
  type: NodePort
  ports:
  - port: 80
    targetPort: 80
    nodePort: 30080
```

#### LoadBalancer
- Provisiona load balancer externo (cloud)
- Integração com AWS ELB, GCP LB, etc.
- IP público automático

#### ExternalName
- Mapeia Service para nome DNS externo
- Útil para serviços externos ao cluster

### Namespaces: Isolamento Lógico

**Namespaces** dividem cluster em ambientes virtuais:

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: producao
---
apiVersion: v1
kind: Namespace
metadata:
  name: desenvolvimento
```

```bash
# Criar recursos em namespace específico
kubectl apply -f deployment.yaml -n producao

# Listar recursos por namespace
kubectl get pods -n desenvolvimento

# Namespace padrão
kubectl config set-context --current --namespace=producao
```

### ConfigMaps e Secrets

#### ConfigMap: Configurações
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  database_url: "postgresql://db:5432/myapp"
  log_level: "info"
  config.properties: |
    server.port=8080
    server.host=0.0.0.0
```

#### Secret: Dados Sensíveis
```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
type: Opaque
data:
  username: YWRtaW4=  # base64 encoded
  password: MWYyZDFlMmU2N2Rm  # base64 encoded
```

#### Usando em Pods
```yaml
spec:
  containers:
  - name: app
    image: minha-app
    env:
    - name: DATABASE_URL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: database_url
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: app-secrets
          key: password
```

### Comandos kubectl Essenciais

#### Gerenciamento de Recursos
```bash
# Aplicar configuração
kubectl apply -f deployment.yaml

# Criar recurso imperativo
kubectl create deployment nginx --image=nginx

# Listar recursos
kubectl get pods
kubectl get deployments
kubectl get services

# Descrever recurso
kubectl describe pod nginx-pod

# Ver logs
kubectl logs nginx-pod

# Executar comando no Pod
kubectl exec -it nginx-pod -- bash

# Deletar recurso
kubectl delete deployment nginx
```

#### Debugging e Troubleshooting
```bash
# Ver eventos do cluster
kubectl get events

# Port forward para acesso local
kubectl port-forward pod/nginx-pod 8080:80

# Copiar arquivos
kubectl cp nginx-pod:/etc/nginx/nginx.conf ./nginx.conf

# Ver uso de recursos
kubectl top nodes
kubectl top pods
```

### Volumes e Persistência

#### EmptyDir: Volume Temporário
```yaml
spec:
  containers:
  - name: app
    volumeMounts:
    - name: cache
      mountPath: /tmp/cache
  volumes:
  - name: cache
    emptyDir: {}
```

#### PersistentVolume e PersistentVolumeClaim
```yaml
# PersistentVolume
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-storage
spec:
  capacity:
    storage: 10Gi
  accessModes:
  - ReadWriteOnce
  hostPath:
    path: /data/pv-storage

---
# PersistentVolumeClaim
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-storage
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
```

## 2. Questões de Múltipla Escolha

### Questão 1
Qual é a menor unidade deployável no Kubernetes?

a) Container
b) Pod
c) Deployment
d) Service

**Gabarito: B**

**Justificativa:** O Pod é a menor unidade deployável no Kubernetes. Um Pod pode conter um ou mais containers que compartilham rede e volumes, sendo sempre co-localizados no mesmo nó. Você não pode deployar containers diretamente - eles sempre fazem parte de um Pod. Deployments e Services são objetos de nível superior que gerenciam Pods.

### Questão 2
Qual componente do Control Plane é responsável por decidir em qual nó um Pod deve ser executado?

a) API Server
b) etcd
c) Scheduler
d) Controller Manager

**Gabarito: C**

**Justificativa:** O Scheduler é responsável por analisar recursos disponíveis nos nós, considerar constraints, affinity rules e outros fatores para decidir onde colocar novos Pods. O API Server processa requisições, etcd armazena dados, e Controller Manager mantém o estado desejado, mas apenas o Scheduler faz decisões de placement.

### Questão 3
Qual tipo de Service você usaria para expor uma aplicação para acesso externo ao cluster com um IP público automático?

a) ClusterIP
b) NodePort
c) LoadBalancer
d) ExternalName

**Gabarito: C**

**Justificativa:** LoadBalancer provisiona automaticamente um load balancer externo (em provedores cloud como AWS, GCP) com IP público, sendo ideal para expor aplicações para internet. ClusterIP é apenas interno, NodePort expõe em portas específicas dos nós (mas sem IP público automático), e ExternalName mapeia para serviços DNS externos.
